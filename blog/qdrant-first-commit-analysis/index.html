Qdrant 첫 커밋 분석 — 벡터 DB는 어떻게 시작되었는가

5,600+ 커밋, 1,100+ Rust 파일로 성장한 Qdrant의 첫 커밋을 분석합니다. 17개 파일에서 시작된 벡터 데이터베이스의 초기 설계 철학을 읽어봅니다.

오픈소스 Qdrant 벡터DB Rust 아키텍처

Qdrant란?

Qdrant는 Rust로 작성된 오픈소스 벡터 데이터베이스입니다. RAG 시스템에서 임베딩 벡터를 저장하고 유사도 검색을 수행하는 핵심 인프라로, Pinecone, Weaviate와 함께 가장 많이 사용되는 벡터 DB 중 하나입니다.

현재 Qdrant는 v1.17.1, 5,628개 커밋, 1,114개 Rust 파일로 구성된 대규모 프로젝트입니다.

그런데 이 프로젝트의 첫 커밋은 어떤 모습이었을까요?

첫 커밋 메타 정보

커밋 해시: e4452500a0228cccbf8dd37e3c9e8f81e8faaf33
날짜:      2020-05-30 23:35:38 +0200
작성자:    Andrey Vasnetsov
메시지:    init

2020년 5월 30일 토요일 밤 11시 35분, Andrey Vasnetsov이 “init”이라는 메시지와 함께 Qdrant의 첫 코드를 커밋했습니다. 주말 밤에 시작된 사이드 프로젝트가 현재 수백만 달러 규모의 벡터 DB 회사가 된 것입니다.

단 17개 파일로 시작합니다:

    │   ├── mod.rs

현재: 1,114개 Rust 파일. 첫 커밋에서 약 65배 성장했습니다.

핵심 분석 1: 타입 정의 — 3줄이 모든 것의 시작

src/common/types.rs는 단 3줄입니다:

pub type PointIdType = u64;
pub type TagIdType = String;
pub type VectorType = Vec<f64>;

이 3줄에서 Qdrant의 핵심 데이터 모델이 보입니다:

  • PointIdType = u64: 벡터 포인트의 고유 ID. 정수형으로 시작 (현재는 UUID도 지원)
  • TagIdType = String: 메타데이터 태그 (이후 “payload”로 진화)
  • VectorType = Vec<f64>: 벡터는 f64 배열. 단순하지만 정확한 표현

설계 관점: 타입 별칭(type alias)을 사용한 것이 눈에 띕니다. 나중에 u64 → UUID로 변경할 때 PointIdType만 수정하면 됩니다. 첫 커밋부터 변경에 열린 설계를 하고 있습니다.

핵심 분석 2: 인덱스 설계 — 처음부터 HNSW를 고려

src/common/index_def.rs:

#[derive(Debug, Deserialize, Serialize)]
pub enum Distance {
  Cosine,
  Euclid
}

#[derive(Debug, Deserialize, Serialize)]
pub struct PlainIndex {
  params: BaseIndexParams,
}

#[derive(Debug, Deserialize, Serialize)]
pub struct HnswIndex {
  params: BaseIndexParams,
  m: usize,
  ef_construct: usize
}

#[derive(Debug, Deserialize, Serialize)]
pub enum IndexType {
  PlainIndex, HnswIndex
}

주목할 점:

  1. Distance 메트릭: Cosine과 Euclid 두 가지를 처음부터 지원. 벡터 검색의 두 핵심 거리 함수를 이미 알고 있었음
  2. PlainIndex vs HnswIndex: 단순 인덱스와 HNSW 인덱스를 enum으로 분리. 아직 구현은 없지만, 인터페이스를 먼저 설계
  3. HNSW 파라미터: m(연결 수)과 ef_construct(빌드 시 탐색 범위) — HNSW 알고리즘의 핵심 파라미터를 정확히 이해하고 있음

핵심 인사이트: 첫 커밋에 구현은 없지만 인터페이스 설계가 정확합니다. “어떻게 만들까”보다 “무엇을 만들까”를 먼저 정의한 것입니다.

핵심 분석 3: Operations — CRUD 설계

포인트 연산 (point_ops.rs)

/// Insert or update points
struct UpsertPoints {
  collection: CollectionUpdateInfo,
  ids: Vec<PointIdType>,
  vectors: Vec<VectorType>
}

/// Delete point if exists
struct DeletePoints {
  collection: CollectionUpdateInfo,
  ids: Vec<PointIdType>,
}

Upsert (Insert or Update)를 처음부터 채택한 것이 인상적입니다. “존재하면 업데이트, 없으면 삽입” — 벡터 DB에서 가장 흔한 패턴을 이미 파악하고 있었습니다.

컬렉션 연산 (collection_ops.rs)

struct CreateCollection {
    collection_name: String,
    dim: usize,
    index: Option<IndexType>,
}

컬렉션 생성 시 **차원(dim)**과 인덱스 타입을 받습니다. index: Option<IndexType> — 인덱스 없이도 컬렉션을 만들 수 있게 한 것은, 데이터를 먼저 넣고 나중에 인덱스를 빌드하는 패턴을 고려한 것입니다.

Alias 시스템

struct CreateAlias {
    collection_name: String,
    alias_name: String,
}

struct RenameAlias {
    old_alias_name: String,
    new_alias_name: String,
}

첫 커밋부터 Alias(별칭) 시스템을 설계했습니다. 이건 프로덕션 환경에서 무중단 인덱스 교체를 위한 패턴입니다 — Elasticsearch의 alias와 동일한 개념. 첫 커밋에서 프로덕션 운영을 고려한 것이 놀랍습니다.

핵심 분석 4: WAL (Write-Ahead Log) — 데이터 안전성

main.rs는 사실상 WAL 테스트입니다:

fn main() {
    test_wal::write_wal();
    test_wal::read_wal();
    test_wal::truncate_wal();
    test_wal::read_wal();
}

첫 커밋에서 가장 먼저 구현한 것이 WAL이라는 점이 중요합니다. 벡터 검색 로직도, API 서버도 아닌 데이터 내구성(durability) 부터 시작했습니다.

docs/wal.md에 적힌 설계 원칙:

  • Once data is submitted successfully, it should not be lost
  • While indexing, power loss may appear in any moment. Index should not be corrupted
  • All submitted data should eventually be available

데이터를 잃지 않는 것이 첫 번째 원칙입니다. 화려한 기능보다 신뢰성을 먼저 구축하겠다는 의지가 보입니다.

핵심 분석 5: 설정 시스템

settings.rs는 계층적 설정 로드를 구현합니다:

// 1. 기본 설정
s.merge(File::with_name("config/config"))?;

// 2. 환경별 설정 (development, production 등)
let env = env::var("RUN_MODE").unwrap_or("development".into());
s.merge(File::with_name(&format!("config/{}", env)).required(false))?;

// 3. 로컬 설정 (git에 커밋하지 않음)
s.merge(File::with_name("config/local").required(false))?;

// 4. 환경 변수 (QDRANT_ 접두사)
s.merge(Environment::with_prefix("QDRANT"))?;

기본값 → 환경별 → 로컬 → 환경변수 순으로 설정을 오버라이드합니다. 12-Factor App 원칙을 따르는 깔끔한 설정 관리입니다.

첫 커밋에 없는 것들

첫 커밋에 없는 것도 중요합니다:

  • 검색 로직: 벡터 유사도 검색 구현이 없음. 데이터 구조와 WAL만 있음
  • API 서버: HTTP/gRPC 서버 없음 (3번째 커밋에서 actix-web 추가)
  • 실제 인덱스 구현: HNSW 인터페이스만 정의, 구현은 없음
  • 테스트: 단위 테스트 없음 (main.rs에서 직접 실행)

이건 나쁜 것이 아닙니다. 필수적인 기반(타입, WAL, 설정)을 먼저 잡고, 기능을 점진적으로 추가하는 전략입니다.

배운 점: 오픈소스 프로젝트의 시작점에서 얻는 교훈

1. 인터페이스를 먼저 설계하라

첫 커밋에 구현은 거의 없지만, 데이터 타입과 연산 인터페이스가 명확합니다. “어떻게 만들까”보다 “무엇이 필요한가”를 먼저 정의했습니다. 이 인터페이스가 이후 5,000+ 커밋의 뼈대가 됩니다.

2. 신뢰성이 기능보다 먼저다

검색 알고리즘도 없는 첫 커밋에서 WAL을 구현한 것은, 데이터를 잃지 않는 것이 벡터 DB의 최우선 과제임을 보여줍니다. 화려한 기능은 나중에 추가할 수 있지만, 데이터 안전성은 처음부터 설계해야 합니다.

3. 타입 별칭으로 변경에 대비하라

pub type PointIdType = u64 — 이 한 줄이 나중에 UUID 지원을 추가할 때 변경 범위를 최소화해줍니다. 작은 결정이지만 프로젝트 전체에 영향을 미치는 좋은 습관입니다.

4. 프로덕션을 처음부터 고려하라

Alias 시스템, 계층적 설정 관리, 환경 변수 지원 — 사이드 프로젝트 수준을 넘어서 프로덕션 운영을 처음부터 고려한 설계입니다.

5. 작게 시작하라

17개 파일, “init” 한 단어의 커밋 메시지. 완벽할 필요 없습니다. 핵심 구조만 잡고 시작하면 됩니다.
